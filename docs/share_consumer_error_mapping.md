# Share Consumer Error Mapping (KIP-932)

How errors in the [KIP-932](https://cwiki.apache.org/confluence/display/KAFKA/KIP-932%3A+Queues+for+Kafka) share consumer surface in this client, and how that lines up with the Java `KafkaShareConsumer`.

**Status:** reflects the `ShareConsumer` on the `dev_kip-932_*` branch (June 2026). The share consumer is experimental and pre-GA, and librdkafka's share-consumer error surface is **not stable yet** — codes and the way they surface may change. Re-verify against the [Sources](#sources) before relying on specifics.

There is **no `DeserializingShareConsumer`** today. `ShareConsumer.poll()` hands back raw `Message` objects carrying `bytes`; there is no key/value deserialization layer, so none of the deserialization-error behavior described for `DeserializingConsumer` applies here. If you're porting from `DeserializingConsumer`, see [Error channels](#error-channels).

## The core difference

Java encodes error semantics in the **exception type**: you `catch (RetriableException e)` or `catch (TopicAuthorizationException e)`, and the class tells you whether it's retriable, an auth problem, fatal, etc.

This client has no such hierarchy. Almost everything from librdkafka becomes a single `KafkaException` wrapping a single `KafkaError`, and the semantics live in **data** on that object — an integer code plus three boolean flags:

```
JAVA                              PYTHON + LIBRDKAFKA
a deep exception class tree   →   one class: KafkaException
                                    └── one arg: KafkaError
                                          ├── .code()              int  ← which error (the tree's leaves)
                                          ├── .name()              str  ← e.g. "SHARE_SESSION_NOT_FOUND"
                                          ├── .str()               str  ← human message
                                          ├── .retriable()         bool ← stands in for RetriableException
                                          ├── .fatal()             bool ← stands in for the "fatal" markers
                                          └── .txn_requires_abort() bool ← transactional producer only
```

The `KafkaError.<NAME>` integer constants are generated by walking librdkafka's `rd_kafka_get_err_descs()`, so **the Python constant name is librdkafka's enum name, not Java's** — see [Name divergences](#name-divergences).

## Catching errors

You collapse Java's many `catch` clauses into one and branch on the code / flags:

```python
from confluent_kafka import KafkaException, KafkaError

try:
    msgs = consumer.poll(timeout)
except KafkaException as e:
    err = e.args[0]                                       # the KafkaError
    if err.code() == KafkaError.TOPIC_AUTHORIZATION_FAILED:
        ...                                               # a specific leaf
    elif err.retriable():                                 # the RetriableException category
        ...                                               # NOTE: only populated on poll/commit/close — see caveats
    elif err.fatal():
        ...
```

## Error channels

Java throws nearly everything out of `poll()`. This client spreads errors across four channels, and the split matters — a `try`/`except` around `poll()` does not catch all of them.

| Channel | Raises? | Carries | Flags populated? |
|---|---|---|---|
| Raised `KafkaException` | yes, aborts the call | closed-handle (`RuntimeError`), bad-arg shape (`TypeError`), config rejection (`ValueError`), and broker/fetch/state errors out of `poll`, `subscribe`, `acknowledge*`, `commit_*`, `close` | only on the `poll`/`commit_*`/`close` subset — see [retriable](#retriable-is-a-flag-not-a-type) |
| `commit_sync()` result dict | no | `Dict[TopicPartition, Optional[KafkaError]]` — per-partition ack outcome (e.g. `INVALID_RECORD_STATE`); `None` value = that partition committed | no (bare code) |
| Ack-commit callback `exception` arg | no | the `callback(offsets, exception)` you registered; `exception` is a `KafkaException` on a rejected commit, else `None` | no (bare code) |
| Per-record `msg.error()` | no, record stays in the list | any per-record error librdkafka attaches to a returned `Message`. Check with `if msg.error():` | n/a |

Two things to internalize:

- **Per-partition acknowledge failures never raise.** They arrive in the `commit_sync()` result dict and/or the ack-commit callback's `exception` argument. A `try`/`except KafkaException` around `commit_sync()` won't see them — read the returned dict. See the [callback section](#the-acknowledgement-commit-callback).
- **`msg.error()` is not a deserialization channel here.** Other (de)serializing clients in this library raise on a bad record (`DeserializingConsumer` → `ValueDeserializationError`, `SerializingProducer` → `ValueSerializationError`, `AvroConsumer` → `SerializerError`). The share consumer has no deserializing variant, so a `try`/`except SerializationError` lifted from those clients will **never fire** — there's nothing to deserialize. `msg.error()` still exists and you should still check it per record, but today it carries only whatever librdkafka stamps on a returned message, not a (de)serialization fault. The one structural fault the binding raises itself is `TypeError` for a non-`Message` argument to `acknowledge()` (the `PyArg_ParseTuple` `O!` check).

## Name divergences

Because the Python constants come from librdkafka's enum rather than Kafka's `Errors`, four names a Java porter would guess differ. Watch for these — a wrong constant name is a silent `AttributeError`, not a type error.

- `NotLeaderOrFollowerException` → `KafkaError.NOT_LEADER_FOR_PARTITION` (not `NOT_LEADER_OR_FOLLOWER`)
- `UnknownTopicOrPartitionException` → `KafkaError.UNKNOWN_TOPIC_OR_PART` (not `…PARTITION`)
- `CorruptRecordException` → `KafkaError.INVALID_MSG` (and `_BAD_MSG` on the share fetch path)
- `InvalidTopicException` → `KafkaError.TOPIC_EXCEPTION`
- `UnknownServerException` → `KafkaError.UNKNOWN`

`InconsistentTopicIdException` (103) used to be in this list as *missing* — librdkafka now defines `INCONSISTENT_TOPIC_ID`, so the constant exists and the names line up.

Internal codes (negative, single-underscore prefix in Python — `KafkaError._TRANSPORT`, `_TIMED_OUT`, `_AUTHENTICATION`, `_BAD_MSG`, `_STATE`, `_CONFLICT`, `_KEY_DESERIALIZATION`, …) are librdkafka/client-internal, not broker protocol. Several Java "leaves" map to these rather than to a broker code.

## "retriable" is a flag, not a type

Java's `RetriableException` is a static base class: the type *is* the contract, and the client auto-retries those internally before surfacing. librdkafka also auto-retries internally (via its per-request action tables), but the **surfaced** retriability is a *per-instance* boolean set by whichever call site builds the error — the same code can be retriable on one error and not another. None of the codes above is "statically retriable" the way the Java tree declares.

More to the point, the flag is only copied off librdkafka's rich `rd_kafka_error_t *` (the `.fatal`/`.retriable`/`.txn_requires_abort` fields), and only some share-consumer calls return that type:

- **Flags populated** (`rd_kafka_error_t *` → `KafkaError_new_from_error_destroy`): `poll`, `commit_sync` (the top-level error), `commit_async`, `close`, `set_acknowledgement_commit_callback`.
- **Flags always `False`** (bare `rd_kafka_resp_err_t` → `KafkaError_new0`, no flag fields): `subscribe`, `unsubscribe`, `acknowledge`, `acknowledge_offset`, **plus the per-partition `KafkaError`s in the `commit_sync` result dict and the `KafkaException` handed to the ack-commit callback**.

So after `poll`/`commit`/`close`, `.retriable()` reflects what librdkafka set; on an ack error, a `commit_sync`-dict error, or a callback error, `.retriable()` and `.fatal()` are always `False` — **branch on `.code()`, not the flags**, on those paths.

## The acknowledgement-commit callback

Registered with `set_acknowledgement_commit_callback(callback)` (pass `None` to clear). This is the Python analog of Java's `KafkaShareConsumer.setAcknowledgementCommitCallback(AcknowledgementCommitCallback)`.

```python
def on_commit(offsets, exception):
    # offsets:   Dict[TopicPartition, set[int]]  — the acked offsets, per partition
    # exception: KafkaException on a rejected commit, else None
    if exception is not None:
        err = exception.args[0]
        if err.code() == KafkaError.INVALID_RECORD_STATE:
            ...   # the broker rejected these acks (e.g. the acquisition lock expired)

sc.set_acknowledgement_commit_callback(on_commit)
```

Maps to Java's `void onComplete(Map<TopicIdPartition, Set<Long>> offsets, Exception exception)`. Differences worth knowing:

- **Offsets key is `TopicPartition`, not `TopicIdPartition`.** No topic UUID is carried yet (same gap as `commit_sync`'s key).
- **Dispatch is on the application thread**, from inside whichever call is draining the response queue — `poll`, `commit_sync`, or `close`. After a `commit_async`, the callback fires on the *next* such call; `commit_sync` and `close` dispatch it inline before they return. It never runs on a background thread.
- **The `exception` is built from a bare code**, so `exception.args[0].retriable()`/`.fatal()` are always `False` — branch on `.code()`.
- **Offsets ride along on the error too.** A rejection doesn't blank out `offsets`; you still get the partition→offsets map, plus the `exception`.
- **Reentrancy is blocked.** Calling any `ShareConsumer` method (including the setter itself) from inside the callback raises `KafkaException(_STATE)` — librdkafka guards every entry point.
- **`commit_sync` reports a rejection twice**, and the two agree: once as the per-partition `KafkaError` in its result dict, once via the callback's `exception` (fired inline).

In the integration suite, `INVALID_RECORD_STATE` (forced by letting a record's acquisition lock lapse, then committing the now-stale ack) is the only rejection reproducible without a mock cluster.

## Provisional error → exception mapping

There is a `TODO KIP-932` in `ShareConsumer.c` proposing that, longer term, codes map to the Python exception a Java porter would expect: `_INVALID_ARG → ValueError` (Java `IllegalArgumentException`), `_STATE → RuntimeError` (Java `IllegalStateException`). Today everything librdkafka returns is a `KafkaException`. Findings from checking that proposal against the Java source and the rest of this client:

- **It would be the first place in this client where a librdkafka *code* becomes anything other than `KafkaException`.** Today `ValueError`/`RuntimeError`/`TypeError` are used only for Python-level validation done *before* calling librdkafka. The regular `Consumer` even manufactures a `KafkaException(_INVALID_ARG)` for a bad argument it detects itself, rather than raising `ValueError`. Adopting the mapping for the share consumer alone makes the two consumers diverge for the same code.
- **A blanket code → class table can't match Java**, because librdkafka collapses many distinct conditions into one code while Java picks its exception per-condition. `_STATE` alone covers closed / not-subscribed / wrong-ack-mode / not-in-flight / reentrancy. Java throws `IllegalStateException` for those — so `_STATE → RuntimeError` holds up — but Java's `acknowledge` throws `IllegalStateException`, never `IllegalArgumentException`, so `_INVALID_ARG → ValueError` does *not* hold on the acknowledge path.
- **Per-partition acknowledge errors should stay `KafkaError`.** `commit_sync` already returns `Dict[TopicPartition, Optional[KafkaError]]`, mirroring Java's `Map<TopicIdPartition, Optional<KafkaException>>` (where `InvalidRecordStateException` is itself a `KafkaException`). Translating them would diverge from Java.

Practical reading: if this is ever implemented, do it per call site (or pre-validate in Python and raise the native exception before calling librdkafka, as the regular `Consumer` already does), not as a global code → class table — and wait for librdkafka to emit distinct codes for the conditions that currently share `_STATE` / `_INVALID_ARG`.
